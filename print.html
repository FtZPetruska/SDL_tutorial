SDL Tutorial

Introduction

This tutorial will show you how to set up SDL2 and its subprojects on your OS and toolchain of choice.

What this tutorial is for

The aim of this tutorial is to provide SDL2 users a guide for their platform and tools. Whether you are using Makefiles on a Unix system with pkg-config, invoking your compiler straight from bash or a fully integrated IDE such as Microsoft Visual Studio on Windows or Xcode on macOS, you should be covered by this guide one way or another.

Some chapters will try to be as cross-platform as possible, meaning that you should be able to take that project and run it on Windows, macOS and Linux without touching it. This is achieved by using CMake and vcpkg

When possible, the other SDL2 subprojects (Image, Mixer, Net, Sound, TTF) will be included.

What this tutorial is not for

This guide will not cover cross-compilation in any way. It will assume that when you are targeting your own machine.

Similarly, this guide assumes a fairly modern computer, capable of running Windows 10, macOS 10.15 or an up-to-date Linux distro.

SDL1.2 will not be covered either, its last release was over ten years ago and has been superseded by SDL2.

Lastly, this will not teach you how to use any of the libraries, some code examples will include some code from them to ensure everything worked but shall remain terse.

Reporting issues

Something does not work for you ? What you see seems outdated ? Feel free to open an issue!

You may also get a quicker answer to your exact problem on the SDL discord.

CMake and VCPKG in Visual Studio Code

This chapter will cover a solution that should work regardless of your OS.

This solution uses:

  • Ninja as a build system. (MSBuild on Windows)
  • CMake to generate build system files.
  • Microsoft VCPKG to grab all the libraries.
  • Visual Studio Code to write the code and interface with CMake

It should be noted that while this solution works on Windows, macOS and Linux it is not necessarily the best solution. For example, if you are on Windows, Microsoft Visual Studio provides an overall better experience and supports all the aforementioned tools, similarly if you have access to a JetBrains CLion license, you may prefer using it instead.

Installation of the Tools

Visual Studio Code binaries can be grabbed on this website. If your Linux distribution does not support the formats offered, it is also available as a flatpak on flathub.

The installation of the other tools is OS-specific:

  • Windows instructions
  • macOS instructions
  • Linux instructions

Windows Installation

In this section, we will only cover the compiler Microsoft provides with Visual Studio and the LLVM toolchain which depends on Visual Studio. If you are only running Windows and your PC is not a potato, I strongly encourage you to use Visual Studio instead of Visual Studio Code as it provides on its own all the functionalities that we are installing separately in this section and overall an overall better developer experience.

Visual Studio

You want to head to Microsoft's website and download Visual Studio 2022 Community.

During the installation you want to pick the "C++ Desktop development" workflow. You want to make sure that the optional feature "MSVC v143 - VS 2022 C++ x64/x86 Build Tools (Latest version)" is ticked on.

Command-Line Tools

In this section, we will use Chocolatey to install CMake and git. Optionally, we will also install the LLVM toolchain and Ninja.

First, head over to Chocolatey's website and follow their install instructions.

To install packages with chocolatey, you will need administrator privileges. You can install a sudo-like alternative using:

> choco install gsudo

This command needs to be entered in an elevated prompt. For the rest of the commands, I will assume that this package is installed and start the commands with sudo when elevation is required.

You can now install CMake and git by running:

> sudo choco install cmake git

Finally, if you want to use the LLVM toolchain, you need to install Ninja along with it:

> sudo choco install llvm ninja

Make sure to reboot

VCPKG

First start by cloning the VCPKG repository and enter the folder:

> git clone https://github.com/Microsoft/vcpkg.git
> cd vcpkg

Then run the following command to download (or compile) the executable, you pass the -disableMetrics argument if you wish disable usage metrics:

> ./bootstrap-vcpkg.bat

Finally, you want to modify your environment variables and add:

VCPKG_ROOT              C:/Path/To/VCPKG
VCPKG_DEFAULT_TRIPLET   x64-windows

And update your PATH by adding a new entry containing: %VCPKG_ROOT%

NB: VCPKG can be on any drive, not necessarily on the C drive.

MacOS Installation

The first thing you will need to install on macOS is command-line developer tools. Even if you are using Xcode tools, they will be needed to install the Homebrew package manager.

To install these tools, open a terminal and run:

% xcode-select --install

Once the command-line tools are installed, you can run this command without root privileges to install Homebrew:

% /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Once the installation finishes, make sure to close and reopen your terminal so the environment gets set properly.

Optional homebrew packages that you may be interested in:

  • git and git-gui as Apple's versions tend to be slightly out of date.
  • llvm for LLVM's version of Clang.
  • gcc for the GNU C/C++ compilers.
  • clang-format for a C/C++ code formatter.

Ninja and CMake

You can install both packages at once using this command:

% brew install cmake ninja

Add the following lines in your .zshrc (or equivalent if you use a different shell):

export CMAKE_EXPORT_COMPILE_COMMANDS=ON
export CMAKE_GENERATOR=Ninja

VCPKG

First start by cloning the VCPKG repository and enter the folder:

% git clone https://github.com/Microsoft/vcpkg.git
% cd vcpkg

Then run the following command to download (or compile) the executable, you pass the -disableMetrics argument if you wish disable usage metrics:

% ./bootstrap-vcpkg.sh

Finally, you want to add the following lines in your ~/.zshrc:

export VCPKG_ROOT=/path/to/vcpkg
export PATH="$VCPKG_ROOT:$PATH"

If your device is running an M1 CPU:

export VCPKG_DEFAULT_TRIPLET=arm64-osx

If your device is running an Intel CPU:

export VCPKG_DEFAULT_TRIPLET=x64-osx

Notes

The LLVM suite of tools will not be added to your path by default as it overlaps with Apple's version (except for clang-format), you will manually need to create the kit entry later on if you wish to use it with that setup.

As the time of writing, vcpkg does not support "universal binaries", and while it is technically possible to define a custom "triplet" to create them, some dependencies of SDL2 subprojects explicitly forbid universal binaries in their CMakeLists.txt. As such only native builds will be covered.

Linux Installation

The package names in this section will only cover Arch Linux and Ubuntu. If a package is not found, please check your own distribution package list before filing an issue.

Ubuntu

Before anything, you want to ensure your system is up to date by running:

$ sudo apt update && sudo apt upgrade

We will first need to install a compiler. The two main options are GCC and LLVM Clang:

$ sudo apt install gcc
$ sudo apt install clang

We then need to install the dependencies for VCPKG:

$ sudo apt install cmake ninja-build git zip unzip tar curl

Finally, we need some specific -dev libraries that are required to build SDL2.

$ sudo apt install libx11-dev libxft-dev libxext-dev pkg-config

You may also be interesting in installing clang-format, which provides a code formatter and clang-tidy which provides a static analyser:

$ sudo apt install clang-format clang-tidy

Arch Linux

Before anything, you want to ensure your system is up to date by running:

$ sudo pacman -Syu

We will first need to install a compiler. The two main options are GCC and LLVM Clang:

$ sudo pacman -S gcc
$ sudo pacman -S clang

We then need to install the dependencies for VCPKG:

$ sudo pacman -S cmake ninja git zip unzip tar curl

Finally, we need some specific libraries that are required to build SDL2.

$ sudo pacman -S libx11 libxft libxext pkg-config

VCPKG

First start by cloning the VCPKG repository and enter the folder:

% git clone https://github.com/Microsoft/vcpkg.git
% cd vcpkg

Then run the following command to download (or compile) the executable, you pass the -disableMetrics argument if you wish disable usage metrics:

% ./bootstrap-vcpkg.sh

Finally, you want to add the following lines in your shell's RC file (~/.bashrc on bash, ~/.zshrc on zsh...):

export VCPKG_ROOT=/path/to/vcpkg
export PATH="$VCPKG_ROOT:$PATH"
export VCPKG_DEFAULT_TRIPLET=x64-linux

Configuring Visual Studio Code

In this section, we will go through the extensions needed in order to use CMake and vcpkg and get syntax highlighting in C/C++.

Extensions

Head over the extension menu to the left, you want the following extensions:

  • ms-vscode.cpptools, provides C/C++ support
  • ms-vscode.cmake-tools, provides CMake commands
  • twxs.cmake, provide CMakeLists.txt syntax highlighting, it is planned to be integrated in cmake-tools version 1.11

Settings

Now that the extensions are installed, press CTRL-SHIFT-P (CMD-SHIFT-P on macOS) and type Preferences: Open Settings (JSON).

You can then add the following entries:

{
  // Previous settings here

  // C/C++
  "C_Cpp.default.cStandard": "c17",
  "C_Cpp.default.cppStandard": "c++20",
  "C_Cpp.default.configurationProvider": "ms-vscode.cmake-tools",

  // CMake
  "cmake.configureOnOpen": false,
  "cmake.generator": "Ninja" // Remove this option on Windows
}

Configuring CMake Kits

For this part, you will need to open a folder with an existing CMakeLists.txt in order for the extension to show up.

Start by opening an empty folder, pressing CTRL-SHIFT-P (CMD-SHIFT-P on macOS) and type CMake: Quick Start, a name for the project, and select Executable.

Once the project gets created, you will start seeing new options in the blue status bar at the bottom, and will also be prompted to select a "kit" (you may need to [Scan for kits] first).

If at this step no kit appears even after scanning, ensure that you have installed a compiler.

We now want to add a new kit that will make use of vcpkg. Start by opening the kits' file by pressing CTRL-SHIFT-P (CMD-SHIFT-P on macOS) and typing CMake: Edit User-Local CMake Kits.

Find the scanned kit using the compiler you want to use and make a copy of it. Note that if you edit a scanned kit, your modifications may be erased if you rescan kits later.

Edit the kit by changing its name (for example by adding (vcpkg) at the end of it) and adding a new field:

"toolchainFile": "[/path/to/vcpkg]/scripts/buildsystems/vcpkg.cmake",

With the new kit saved, on the status bar, click on No Active Kit to select your new kit, then on the build right next to the kit to ensure the configuration and build works.

Installing packages with VCPKG

There are two ways to use VCPKG, the first one includes manually using the vcpkg install command to manually install the packages, the second is to fill a project-specific vcpkg.json manifest that lists all the desired dependencies.

Classic mode

Open a terminal and run the following command:

% vcpkg search sdl2

This command will list all the available SDL2 libraries and their optional features.

If you want SDL2_image with libjpeg-turbo and libwebp support you would run:

% vcpkg install "sdl2-image[libjpeg-turbo,libwebp]"

For this tutorial, we will not be using any optional features, so here is the command used:

% vcpkg install sdl2 sdl2-image sdl2-mixer sdl2-net sdl2-ttf

NB: At the time of writing, SDL2_sound is not available on VCPKG.

Finally, make the packages available user-wide with the command:

% vcpkg integrate install

In classic mode, the packages will not be updated automatically. Make sure to go in VCPKG's folder once in a while, run git pull to get the latest changes, bootstrap a new executable, and run vcpkg update to check for package updates.

Manifest mode

Create a file called vcpkg.json next to the CMakeLists.txt, and write the following:

{
  "$schema": "https://raw.githubusercontent.com/microsoft/vcpkg/master/scripts/vcpkg.schema.json",
  "name": "my-application",
  "version": "0.0.1",
  "dependencies": ["sdl2", "sdl2-image", "sdl2-mixer", "sdl2-net", "sdl2-ttf"]
}

VCPKG will automatically pick up the manifest and grab the dependencies during CMake configure stage and make them available to this project.

Similarly to classic mode, you can specify features for your dependencies in the manifest. Here's what it would look like to only add support for libjpeg-turbo and libwebp:

"dependencies": [
  {
    "name": "sdl2-image",
    "default-features": false,
    "features": [
      "libjpeg-turbo",
      "libwebp"
    ]
  }
]

Using the libraries in the build system

In this section, we will see how to tell CMake which libraries we need and how to link them.

Finding the libraries

After VCPKG finished compiling the dependencies, it should output something similar to:

The package sdl2 provides CMake targets:

    find_package(SDL2 CONFIG REQUIRED)
    target_link_libraries(main PRIVATE SDL2::SDL2 SDL2::SDL2main SDL2::SDL2-static)

The package sdl2-image provides CMake targets:

    find_package(sdl2-image CONFIG REQUIRED)
    target_link_libraries(main PRIVATE SDL2::SDL2_image)

The package sdl2-mixer provides CMake targets:

    find_package(sdl2-mixer CONFIG REQUIRED)
    target_link_libraries(main PRIVATE SDL2::SDL2_mixer)

The package sdl2-net provides CMake targets:

    find_package(sdl2-net CONFIG REQUIRED)
    target_link_libraries(main PRIVATE SDL2::SDL2_net)

The package sdl2-ttf provides CMake targets:

    find_package(sdl2-ttf CONFIG REQUIRED)
    target_link_libraries(main PRIVATE SDL2::SDL2_ttf)

This roughly explains how to find and use each package.

To get started, you can edit the CMakeLists.txt generated using the CMake: Quick Start command.

Before the add_executable statement, you want to add the following lines:

find_package(SDL2 CONFIG REQUIRED)
find_package(sdl2-image CONFIG REQUIRED)
find_package(sdl2-mixer CONFIG REQUIRED)
find_package(sdl2-net CONFIG REQUIRED)
find_package(sdl2-ttf CONFIG REQUIRED)

This will tell CMake to look for all the libraries, and that they are required (configuration will fail if they are not found).

After the add_executable statement, you want to add the following lines:

target_link_libraries(
  Tutorial
  PRIVATE
  SDL2::SDL2
  SDL2::SDL2main
  SDL2::SDL2_image
  SDL2::SDL2_mixer
  SDL2::SDL2_net
  SDL2::SDL2_ttf)

This will tell CMake to link all the libraries to your application.

Your CMakeLists.txt should look like this:

cmake_minimum_required(VERSION 3.0.0)
project(Tutorial VERSION 0.1.0)

find_package(SDL2 CONFIG REQUIRED)
find_package(sdl2-image CONFIG REQUIRED)
find_package(sdl2-mixer CONFIG REQUIRED)
find_package(sdl2-net CONFIG REQUIRED)
find_package(sdl2-ttf CONFIG REQUIRED)

add_executable(Tutorial main.cpp)

target_link_libraries(
  Tutorial
  PRIVATE
  SDL2::SDL2
  SDL2::SDL2main
  SDL2::SDL2_image
  SDL2::SDL2_mixer
  SDL2::SDL2_net
  SDL2::SDL2_ttf
)

Testing it

Firstly, you should run the configure stage with CMake, this can be done by pressing CTRL-SHIFT-P (CMD-SHIFT-P on macOS) and running CMake: Configure. With the right kit selected, the configuration step should finish without error.

You can then try to build using the build button on the status bar, this should compile the default main.cpp file (a simple hello world) and link it against all the libraries we have downloaded.

Lastly, we will replace the content of main.cpp to initialize all the libraries, print out their version and shut them all down, ensuring everything worked correctly.

The code in question 
#include <SDL.h>
#include <SDL_image.h>
#include <SDL_mixer.h>
#include <SDL_net.h>
#include <SDL_ttf.h>

/**
 * @brief Initializes every library.
 */
void init_all(void) {
  if (0 > SDL_Init(SDL_INIT_VIDEO)) {
    SDL_Log("%s\n", SDL_GetError());
    return;
  }

  if (0 > IMG_Init(IMG_INIT_PNG)) {
    SDL_Log("%s\n", IMG_GetError());
    return;
  }

  if (0 > Mix_Init(MIX_INIT_MP3)) {
    SDL_Log("%s\n", Mix_GetError());
    return;
  }

  if (0 > SDLNet_Init()) {
    SDL_Log("%s\n", SDLNet_GetError());
    return;
  }

  if (0 > TTF_Init()) {
    SDL_Log("%s", TTF_GetError());
    return;
  }

  SDL_Log("Every library was initialized successfully.\n");
}

/**
 * @brief Quits all libraries.
 */
void quit_all(void) {
  TTF_Quit();
  SDLNet_Quit();
  Mix_Quit();
  IMG_Quit();
  SDL_Quit();
}

/**
 * @brief For each libraries, logs the installed version.
 */
void log_libraries_version(void) {
  SDL_version sdl_compiled_version;
  SDL_VERSION(&sdl_compiled_version);
  SDL_version sdl_linked_version;
  SDL_GetVersion(&sdl_linked_version);
  SDL_Log("SDL:\n"
          "\tCompiled against: %d.%d.%d\n"
          "\tLinked against: %d.%d.%d\n",
          sdl_compiled_version.major, sdl_compiled_version.minor,
          sdl_compiled_version.patch, sdl_linked_version.major,
          sdl_linked_version.minor, sdl_linked_version.patch);

  SDL_version img_compiled_version;
  SDL_IMAGE_VERSION(&img_compiled_version);
  const SDL_version *img_linked_version = IMG_Linked_Version();
  SDL_Log("SDL Image:\n"
          "\tCompiled against: %d.%d.%d\n"
          "\tLinked against: %d.%d.%d\n",
          img_compiled_version.major, img_compiled_version.minor,
          img_compiled_version.patch, img_linked_version->major,
          img_linked_version->minor, img_linked_version->patch);

  SDL_version mix_compiled_version;
  SDL_MIXER_VERSION(&mix_compiled_version);
  const SDL_version *mix_linked_version = Mix_Linked_Version();
  SDL_Log("SDL Mixer:\n"
          "\tCompiled against: %d.%d.%d\n"
          "\tLinked against: %d.%d.%d\n",
          mix_compiled_version.major, mix_compiled_version.minor,
          mix_compiled_version.patch, mix_linked_version->major,
          mix_linked_version->minor, mix_linked_version->patch);

  SDLNet_version net_compiled_version;
  SDL_NET_VERSION(&net_compiled_version);
  const SDLNet_version *net_linked_version = SDLNet_Linked_Version();
  SDL_Log("SDL Net:\n"
          "\tCompiled against: %d.%d.%d\n"
          "\tLinked against: %d.%d.%d\n",
          net_compiled_version.major, net_compiled_version.minor,
          net_compiled_version.patch, net_linked_version->major,
          net_linked_version->minor, net_linked_version->patch);

  SDL_version ttf_compiled_version;
  TTF_VERSION(&ttf_compiled_version);
  const SDL_version *ttf_linked_version = TTF_Linked_Version();
  SDL_Log("SDL TTF:\n"
          "\tCompiled against: %d.%d.%d\n"
          "\tLinked against: %d.%d.%d\n",
          ttf_compiled_version.major, ttf_compiled_version.minor,
          ttf_compiled_version.patch, ttf_linked_version->major,
          ttf_linked_version->minor, ttf_linked_version->patch);
}

int main(int, char **) {
  init_all();
  log_libraries_version();
  quit_all();
}

After pasting the code, you can simply hit build and run the executable, if no error shows up: congratulations, you are ready to get coding!

Notes about SDL2main

When coding an app, you may not care about arguments passed through command line, and may as such be tempted to change your entry point to:

int main()

This declaration is unfortunately incompatible with the way SDL2main works. If you do not care about arguments, and have warnings turned on for unused variables, you have two ways to ignore them:

int main(int, char **) { // C++-only
  // Your code
}

int main(int argc, char *argv[]) { // C/C++
  (void)argc;
  (void)argv;
  // Your code
}

The first option consists in telling the compiler: "Yes, there are two arguments of these types in my signature, but I will not use them, and as such did not give them a name.” The second option is to add a no-op statement for each argument, so they are considered "used" despite not doing anything.
